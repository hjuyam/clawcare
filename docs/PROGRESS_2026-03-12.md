- **M1（安全基线）已完成并测试全绿**：Auth/Session、RBAC（admin/viewer）、Safe Mode、Policy deny + 审计（best-effort）、Ops/Config/Security 页面壳与 API 骨架。

## 1) 已完成能力清单（按 M1/M2/M3/M4 分组）

### M1 — 安全基线（已完成 ✅）
依据：`docs/STATUS.md`、`docs/FEATURES_M1.md`、`docs/TEST_REPORT_M1.md` + 代码扫描

- Auth/Session
  - `/login`
  - `POST /api/auth/login`、`POST /api/auth/logout`
- RBAC（最小可用）
  - 角色：`admin` / `viewer`
  - viewer 只读；admin 在非 Safe Mode 下可触发高危动作
- Safe Mode
  - Safe Mode 开关与拦截逻辑（Safe Mode 下即使 admin 也会被阻断高危操作）
- Policy deny + 审计（best-effort）
  - 策略拒绝（未登录/权限不足/Safe Mode/confirm 缺失）返回明确 deny
  - 审计事件写入 `data/audit/events.jsonl`（敏感字段脱敏）
- Ops / Config / Security 的最小界面壳
  - `/ops`、`/config`、`/security` 页面存在
  - Ops 提供自检/重启/清理/诊断包等 API 骨架并纳入策略约束
- 测试与验收（全绿）
  - Vitest：`12 passed`
  - Playwright：`8 passed`（以 M1 安全合约为主）

### M2 — Runs/Tasks 最小闭环（已完成 ✅，可验收）
依据：`docs/STATUS.md`、`docs/TEST_REPORT_M2.md` + 代码扫描（`app/tasks/*`、`app/api/runs/*`、`app/ops/_components/CreateRunCard.tsx`）

- Runs API
  - `POST /api/runs`：创建 run（按 type + risk 规则受 RBAC/Safe Mode 约束）
  - `GET /api/runs?limit=50`：列表
  - `GET /api/runs/:id`：详情
- Runs 存储（best-effort）
  - 文件持久化：`data/runs.json`
  - 采用“写临时文件再 rename”的 atomic-ish 写入
  - 读失败/JSON 解析失败按空存储容错（best-effort）
- Runs UI
  - `/tasks`：最近 50 条 runs 列表
  - `/tasks/[id]`：run 详情（JSON dump）
- Ops → Runs 的最小入口
  - `/ops` 页面提供“创建 Run”卡片（`ops.self_check` / `ops.restart_gateway` / `ops.cleanup` / `ops.diagnostics_bundle`）
- 执行器（mock executor）
  - `scheduleMockExecution` 以 best-effort timer 将 run 从 queued → running → succeeded
  - **不会真正执行 ops/config 逻辑**（现阶段仅闭环状态流转）
- 测试（全绿）
  - Vitest：`15 passed`
  - E2E：仍以 M1 安全合约为主（runs UI 合约待补）

### M3 — Config Center（部分具备 ⚠️，仍属雏形）
依据：`docs/STATUS.md` 规划项 + 代码扫描（`app/config/*`、`app/api/config/*`）

已具备：
- Config 读取与版本清单
  - `GET /api/config/current`：返回 current_version + etag + config
  - `GET /api/config/versions`：返回 manifest entries
- 配置差异预览（masked diff）
  - `POST /api/config/preview_diff`：对 current vs draft 做 diff（并对敏感字段 mask）
- Apply/Rollback 的“排队入口”（走 runs，mock）
  - `POST /api/config/apply`：`confirm=true` 才会排队（并受 admin + Safe Mode 拦截）
  - `POST /api/config/rollback`：`confirm=true` 才会排队（并受 admin + Safe Mode 拦截）
- 前端 UI
  - `/config` 提供 Draft JSON、Preview diff、Apply、Rollback 按钮

仍缺失：
- apply/rollback 对 manifest/snapshots 的**真实写入与回滚**（目前仅创建 run，run 执行仍为 mock）
- “备份→diff→apply/rollback→可回滚”的完整可验证链路（含审计面板）

### M4 — Memory Center（未开始 ❌）
- `docs/STATUS.md` 中规划存在，但当前 main 未见对应 app 页面/API。

## 2) 当前缺口与阻塞（P0/P1/P2）

### P0（阻塞验收/上线风险）
1. **Runs 仅状态闭环，未真正执行 ops/config 动作**
   - 现状：run executor 为 timer mock；创建 `ops.*` / `config.*` run 不会触发实际逻辑。
   - 风险：对外宣称“可追踪运维动作”时，可能被认为是“只做了工单/队列壳”。
2. **Runs UI 的 E2E 合约缺失**
   - 当前 E2E 仍聚焦 M1 安全合约；M2 “创建 run→列表出现→详情可读”未覆盖。

### P1（影响可信度/可运维性，建议尽快补齐）
1. **存储与并发鲁棒性不足**（文件存储 best-effort）
   - 典型风险：多 worker/并发写导致覆盖、丢数据；serverless 下 timer 不可靠。
2. **Security/Audit 页面目前为占位**
   - API 有 `data/audit/events.jsonl`，但缺少可视化、筛选/查询、关键事件视图。
3. **RBAC 角色模型与 Security 页描述不一致**
   - UI 占位提到 Operator/TOTP 等，但实际实现仅 `admin/viewer`，容易造成验收口径分裂。

### P2（体验/可扩展性优化）
1. `/tasks/[id]` 详情页为 JSON dump，可用但不友好；缺少关键字段摘要、输入/输出展示。
2. Config 变更 reason 在 high-risk 情况仅“建议必填/占位”，未强制策略化（可按风险等级强制）。
3. Audit 事件结构齐全，但缺少“diff_summary/before/after ref”在 config 场景的落地。

## 3) 推进计划（以可合并 PR 切片）

### 下一天（1 天内可落地）
**PR-1：补齐 M2 Runs UI 的 E2E 合约（不改业务逻辑）**
- 覆盖：
  - 登录后进入 `/ops` 创建 `ops.self_check` run
  - `/tasks` 列表出现新 run
  - 进入 `/tasks/[id]` 能看到 run JSON（status/created_at/type）
  - RBAC：viewer 创建 high-risk run 返回拒绝（或 UI 展示错误）
  - Safe Mode：admin 创建 high-risk run 被阻断
- 收益：把“可验收闭环”固化为回归合约。

### 下一周（可交付的里程碑增量）
**PR-2：实现最小“真实执行”的 run executor（先从 ops.self_check 开始）**
- 将 `ops.self_check` 从 mock 状态机升级为：run 执行时写入 result（仍可 mock 内容，但链路真实）
- 结构：run type → handler 映射；统一更新 run.status/result/error；对异常写入 audit

**PR-3：Config apply/rollback 的真实落地（基于现有 manifest/snapshots 工具）**
- apply：在 run 执行阶段将 draft config `snapshotConfig`，更新 manifest currentVersion
- rollback：将 currentVersion 指向 target_version（或 snapshot_id，需统一一种）
- 输出：在 run.result 中写入 before/after version；audit 记录 diff_summary（masked）

**PR-4：Audit 面板最小可用（/security 从占位变可读）**
- `GET /api/audit/log`（若已有则复用）读取 events.jsonl
- `/security` 展示最近 N 条事件 + action/resource_type 过滤

**PR-5：Runs 存储与执行环境加固（按部署形态选择其一）**
- 选项 A：引入 SQLite（推荐，用于并发/一致性）
- 选项 B：append-only log + compaction（保持文件形态但更可靠）
- 同时明确：serverless 下 timer mock 的行为边界（或改为显式“手动推进/worker 触发”）
