## 1) 里程碑验收标准（M2 / M3 / M4）

> 约束：每条标准都必须“可测试/可证明”。下面每条都给出了对应证据（命令/页面/API）。

### M2 — Runs/Tasks 最小闭环（v0 可上线）

**M2.A API 合约（可证明）**
1. 具备 Runs API：
   - `POST /api/runs` 创建 run（low-risk 对 viewer 允许；high-risk 需 admin）。
   - `GET /api/runs` 返回列表（默认 limit=50）。
   - `GET /api/runs/:id` 返回详情。
2. Runs 数据持久化：写入 `data/runs.json`（best-effort 但不得导致服务崩溃）。

**验证方式**
- 单测：`npm test`
- 手工：启动 `npm run dev` 后，在 UI `Tasks & Runs` 创建/查看。

**M2.B UI（可证明）**
1. `/tasks` 可查看 run 列表。
2. `/tasks/[id]` 可查看 run 详情（状态、requested_by、reason、input/result/error）。

**M2.C 安全护栏（可证明）**
1. 未登录访问 runs 创建/高危 ops/config 必须被拒绝，并写入审计（best-effort）。
2. Safe Mode 启用时，高危 run 创建被拒绝（例如 `ops.cleanup`）。

**验证方式**
- E2E：`npm run test:e2e`
- 证据文档：`docs/TEST_REPORT_M2.md`

### M3 — Config Center（备份→diff→apply/rollback→审计，且受 RBAC + Safe Mode 约束）

> **M3 的验收重点不是“能改 JSON”**，而是“变更链路有保险丝 + 可追溯 + 可回滚”。

**M3.A API 合约与行为（可证明）**
1. `GET /api/config/current`：
   - 返回 `current_version`、`etag`、`config`。
   - viewer 可访问。
2. `GET /api/config/versions`：
   - 返回 `current_version` + `entries[]`（至少含 version/hash/time/file/author/reason）。
   - viewer 可访问。
3. `POST /api/config/preview_diff`：
   - 输入 `{config, base_version}`，输出 masked diff（敏感字段打码）
   - viewer 可访问。
4. `POST /api/config/apply`：
   - admin-only；必须 `confirm=true`（否则 409 `CONFIRM_REQUIRED`）。
   - Safe Mode 启用时必须拒绝（403/409 皆可，但需要有结构化 error + requestId）。
   - 成功时**创建一条 run**（type=`config.apply`）并返回 `run_id`。
5. `POST /api/config/rollback`：
   - admin-only；必须 `confirm=true`。
   - Safe Mode 启用时必须拒绝。
   - 成功时**创建一条 run**（type=`config.rollback`）并返回 `run_id`。

**验证方式**
- 单测：`npm test`（关键用例已在 `app/api/config/__tests__/*` 覆盖：confirm gate + safe mode）

**M3.B 版本与回滚语义（可证明）**
1. apply 成功后，`/api/config/current` 的 `current_version` 必须前进（或在同一次 run 成功后前进）。
2. 回滚到历史版本后，`/api/config/current` 必须回到目标版本内容。
3. `data/manifest.json` 与 `data/snapshots/*.json`：
   - 每次 apply 必须先产生 “auto-snapshot before apply”。
   - 产生新的版本条目（hash 与 config 内容一致）。

**验证方式**
- 单测补齐（建议在 M3 完成前新增）：
  - apply 后 `current_version` 变化、snapshot 文件存在
  - rollback 后 `current_version` 回退、config 内容一致

**M3.C UI/UX（可证明）**
1. `/config` 页面具备：
   - Current 版本展示 + refresh
   - Draft JSON 编辑
   - Preview diff（masked）
   - Apply（confirm=true）
   - Rollback（至少可回滚到 previous；更优：可选 target）
2. 成功 apply/rollback 后，UI 明确提示 `run_id`，并引导至 `/tasks` 查看执行记录。

**验证方式**
- E2E（建议补齐作为 M3 退出条件）：`npm run test:e2e` 覆盖 config 页基本交互（preview/apply/rollback）。

**M3.D 审计（可证明）**
1. 所有 config 相关动作（preview/apply/rollback/deny）写入审计（best-effort），且包含：
   - actor/session
   - action/resource_type
   - policy_decision/policy_reason（拒绝时必填）
   - reason（高危操作必填；无则给默认值）

**验证方式**
- 单测（可新增一条）：对 `data/audit/events.jsonl` 新增事件条目断言（或通过 audit API 读取断言）。

### M4 — Memory Center（浏览/搜索/清理 + 安全护栏对齐 Safe Mode）

> M4 的验收重点：**可见、可搜、可控（删除/保留）**，并与安全模型（RBAC/Safe Mode/Audit）一致。

**M4.A API（可证明）**
1. `GET /api/memory/summary`（或 `list`）：
   - 返回今日/昨日/近 7 天的 memory 条目列表（最小字段：id、time、source、title/preview）。
   - viewer 可访问。
2. `POST /api/memory/search`：
   - 支持关键词搜索（最小：substring/正则其一），支持 source 过滤（可选）。
   - viewer 可访问。
3. `POST /api/memory/delete`（或 `DELETE /api/memory/:id`）：
   - admin/operator 可删除；viewer 禁止。
   - Safe Mode 启用时必须拒绝。
   - 删除必须写入审计。
4. （可选但强烈建议）`POST /api/memory/pin`：
   - 支持 pin/unpin（不等同删除），用于“保留策略”。

**验证方式**
- 单测：`npm test`（新增 memory API 测试：RBAC + Safe Mode + audit）

**M4.B UI（可证明）**
1. `/memory` 页面不再是占位：
   - 今日/昨日/近 7 天列表（分页/折叠均可）。
   - 搜索框（关键词）。
   - 删除/Pin 操作（根据角色显示/禁用）。
   - 显示“操作已入审计/被 Safe Mode 拒绝”的明确反馈。

**验证方式**
- E2E：`npm run test:e2e`（覆盖：viewer 可浏览/搜索但不可删除；admin 可删除；safe mode 拒绝删除）

## 3) 进度汇报机制（主动汇报、模板、升级路径）

### 3.1 主动汇报节奏（默认）

**固定节奏（UTC）**
- **T+0（现在）**：发布本 PM Plan（本文件）。
- **每天 2 次**：
  - 09:30 UTC：同步“昨日合入/今日计划/阻塞点”。
  - 19:30 UTC：同步“当天产出/未达预期原因/下一步”。

**事件驱动（随时）**
- 任一 PR 合并后：立即汇报（包含 commit/PR 链接 + 风险提示）。
- 测试红了超过 30 分钟：立即汇报并给出 rollback/隔离方案。
- 被阻塞超过 2 小时（等待权限/决策/外部依赖）：立即升级。

### 3.2 汇报模板（复制即用）

> **[ClawCare] 日更（UTC YYYY-MM-DD HH:mm）**

**摘要（1 句话）**：
- 例如：M3 Config Center 已具备 preview/apply/rollback 的 UI+API，正在补齐“apply/rollback 真实落盘版本推进”与 E2E。

**Done（已完成）**
- PRs：
  - #PR-x：xxx（验收：`npm test` ✅ / `npm run test:e2e` ✅）
- 关键决策：
  - 例如：回滚是否产生新版本：选择 A（原因：…）

**In Progress（进行中）**
- PRs：
  - #PR-y：xxx（风险：…）

**Next（下一步 24h）**
- 1) …
- 2) …

**Risks / Blockers（风险/阻塞）**
- 风险：…（影响：…；缓解：…）
- 阻塞：需要 CrayOn 决策/确认：…（截止时间：…）

**Metrics（客观指标）**
- `npm test`：pass/fail
- `npm run test:e2e`：pass/fail
- `npm run build`：pass/fail

### 3.3 失败/阻塞升级规则（避免“卡死不说”）

- **P0 阻塞（立即升级）**：
  - 安全回归（RBAC/Safe Mode 漏洞）
  - 数据损坏风险（manifest/runs/audit 写坏）
  - 生产不可用（`npm run build` 持续失败）
- **P1 阻塞（2 小时升级）**：
  - 需要产品语义决策（例如 rollback 是否写新版本条目）
  - E2E 不稳定且无法在 2 小时内定位
- **P2 阻塞（下一次例会同步）**：
  - 文案/UI 微调
  - 低优先级重构

升级时必须给出 3 件事：
1) 当前现象（含日志/失败用例）
2) 影响面（用户/安全/进度）
3) 两个备选方案（含成本与风险）
